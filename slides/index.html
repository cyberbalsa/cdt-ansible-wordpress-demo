What is Infrastructure Automation?

Instead of manually running commands on each server, we write code that describes the desired state of our systems.

Without Automation

- SSH into each server one by one

- Run commands manually

- Hope you remember the steps

- Repeat for every new server

With Automation

- Write configuration once

- Apply to any number of servers

- Configuration is documented

- Reproducible every time

What is Ansible?

Ansible is an open-source automation tool created by Red Hat. It connects to your servers via SSH and executes tasks you define in YAML files.

  • Agentless - Nothing to install on target servers. Uses existing SSH.
  • Declarative - You describe WHAT you want, not HOW to do it.
  • Idempotent - Running the same playbook twice won't break anything. It only makes changes when needed.
  • YAML-based - Human-readable configuration files, easy to learn.

Our Architecture: 3-Tier Web Application

Internet Users
       |
       v
+------------------+        +------------------+        +------------------+
|   Web Server     |  --->  |  App Server      |  --->  |  Database        |
|     NGINX        |        |  WordPress/PHP   |        |     MySQL        |
|   fffics-srv-4   |        |   fffics-srv-3   |        |   fffics-srv-2   |
|  (reverse proxy) |        | (runs PHP code)  |        | (stores data)    |
+------------------+        +------------------+        +------------------+

This is a common production pattern: separate concerns across multiple servers for security, scalability, and maintainability.

Why Separate Servers?

Security

Database is not directly exposed to the internet. Only the web server is public-facing.

Scalability

Need more capacity? Add more WordPress servers behind the load balancer.

Maintainability

Update WordPress without touching the database. Restart Nginx without affecting PHP.

Fault Isolation

If WordPress crashes, your database keeps running. Problems are contained.

The Inventory File

The inventory tells Ansible which servers exist and how to group them:

# Groups are defined in [brackets]
[webserver]
fffics-srv-4 ansible_host=100.65.5.241

[wordpress]
fffics-srv-3 ansible_host=100.65.4.183

[database]
fffics-srv-2 ansible_host=100.65.6.146

# Variables that apply to ALL servers
[all:vars]
ansible_user=cyberrange

Group names like [database] let us target specific servers in our playbook.

How Ansible Connects: SSH

SSH (Secure Shell) is a protocol for securely connecting to remote computers over a network.

Password Authentication

Type password every time you connect. Less secure, tedious for automation.

Key Authentication

Use a cryptographic key pair. More secure, no password needed.

We use SSH keys so Ansible can connect to all 3 servers without prompting for passwords.

Step 1: Set Up SSH Keys

First, update your server IPs in inventory.ini, then run:

./setup-ssh.sh

This script will:

  • Generate an SSH key pair if you don't have one
  • Copy your public key to all 3 servers (so they trust your machine)
  • Test that passwordless SSH works

Important: Update the IP addresses in inventory.ini to match YOUR servers before running!

Step 2: Test Ansible Connectivity

The ping module tests if Ansible can reach and authenticate to your servers:

ansible all -m ping

Breaking this down:

  • ansible - the command-line tool
  • all - target all servers in the inventory
  • -m ping - use the "ping" module (not network ping - this tests SSH + Python)
fffics-srv-2 | SUCCESS => { "ping": "pong" }
fffics-srv-3 | SUCCESS => { "ping": "pong" }
fffics-srv-4 | SUCCESS => { "ping": "pong" }

Anatomy of an Ansible Task

Tasks are the building blocks. Each task calls a module with parameters:

- name: Install MySQL server
  ansible.builtin.apt:
    name: mysql-server
    state: present
    update_cache: true
- name: Human-readable description (shows in output when running)
ansible.builtin.apt: Fully qualified module name (FQCN)
name: Module parameter: which package to install
state: present Module parameter: ensure package IS installed

Understanding YAML Syntax

YAML is a human-readable data format. Key rules:

# Key-value pairs
name: mysql-server
state: present

# Lists (start with -)
packages:
  - nginx
  - php
  - mysql

# Nested structures
ansible.builtin.apt:
  name: nginx
  state: present

Indentation matters!

Use spaces, never tabs. 2 spaces per level is standard.

Use true/false not yes/no

VS Code highlights YAML syntax errors in red.

Using Variables

Variables let you define values once and reuse them everywhere:

# Defined in group_vars/all.yml
mysql_root_password: "StudentDemo123!"
wordpress_db_name: "wordpress"

# Used in a task (note the curly braces and quotes)
- name: Create WordPress database
  community.mysql.mysql_db:
    name: "{{ wordpress_db_name }}"
    login_password: "{{ mysql_root_password }}"

Why variables? Change the password in one file, and every task using it updates automatically. Essential for maintaining large configurations.

Common Ansible Modules (FQCN)

Module Purpose
ansible.builtin.apt Install/remove packages on Debian/Ubuntu
ansible.builtin.service Start/stop/enable system services
ansible.builtin.file Create files/directories, set permissions
ansible.builtin.template Copy files with variable substitution
ansible.builtin.get_url Download files from URLs
community.mysql.mysql_db Create/manage MySQL databases

FQCN = Fully Qualified Collection Name (best practice for Ansible 2.10+)

Templates: Dynamic Configuration Files

Templates are files with placeholders that Ansible fills in with variable values:

# Template file: wp-config.php.j2
define('DB_HOST', '{{ database_server }}');
define('DB_NAME', '{{ wordpress_db_name }}');
define('DB_USER', '{{ wordpress_db_user }}');

# After Ansible processes it, becomes:
define('DB_HOST', '100.65.6.146');
define('DB_NAME', 'wordpress');
define('DB_USER', 'wp_user');

The .j2 extension means "Jinja2 template" - the templating language Ansible uses.

What is a Reverse Proxy?

Nginx sits in front of WordPress, receiving all requests and forwarding them:

User Request --> NGINX (port 80) --> WordPress/PHP (port 9000)
                   |
                   +-- Handles: SSL, caching, load balancing, security

Benefits:

  • Security: WordPress never directly exposed to internet
  • Performance: Nginx handles static files efficiently
  • Flexibility: Can add caching, SSL, or multiple backends later

Running the Full Playbook

ansible-playbook playbook.yml

What happens:

  • Ansible reads the playbook and inventory
  • Connects to each server via SSH
  • Executes tasks in order, one server group at a time
  • Reports success (green), changes (yellow), or failures (red)

Idempotent: Running the playbook again will only report "ok" - no unnecessary changes.
